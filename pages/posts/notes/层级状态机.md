## 1. 引言

在嵌入式业务中，只要流程足够长，状态机最终都会出现；典型过程通常并不复杂：初始化、等待指令、准备目标、执行动作、等待反馈、处理异常、结束复位；真正让代码逐渐失控的，不是单个阶段本身，而是以下几类问题：

- 单一的 `switch-case` 逻辑混乱，难以维护
- 业务分支越来越多，状态跳转散落在各处
- 事件处理堆积在一个函数里，导致代码臃肿
- 错误处理被重复写在多个分支里
- 一次性动作与周期轮询混在一起
- 后续新增阶段时，原有流程越来越难改

基于 C 模拟 OOP 的层级有限状态机（HFSM）是一种非常有效的收口手段，核心思路即：

- 把任务流程中的每个阶段抽象成“状态”
- 用 `handle` 处理离散事件
- 用 `entry` / `exit` 处理状态切换时的一次性动作
- 用 `action` 承担状态内的周期推进
- 用层级关系组织状态，把公共规则上提到父状态统一处理

## 2. FSM 到 HFSM

### 2.1. 最简单的 FSM

一个最简单的 FSM 可能长这样，由一个大的 `switch-case` 语句组成：

```c
switch(state) {
    case STATE_INIT:
        if(init_ok) {
            state = STATE_IDLE;
        }
        break;

    case STATE_IDLE:
        if(start_cmd) {
            state = STATE_PREPARE;
        }
        if(error_flag) {
            state = STATE_FAULT;
        }
        break;

    case STATE_PREPARE:
        if(target_ready) {
            state = STATE_RUN;
        }
        if(error_flag) {
            state = STATE_FAULT;
        }
        break;

    case STATE_RUN:
        if(task_done) {
            state = STATE_FINISH;
        }
        if(error_flag) {
            state = STATE_FAULT;
        }
        break;
}
```

在流程很短的时候，这种写法直观且有效。但一旦业务开始扩展，问题会很快出现：

- 各状态都在重复判断同一种异常
- 进入状态时的一次性动作和周期轮询逻辑混在一起
- 某些状态转移需要跨多个分支才能看完整
- 状态数量一多，整体结构迅速失去可读性
- 新增一个阶段时，往往需要在很多地方补逻辑

问题的根源是**所有职责都被塞进了一个平面结构里**，普通 FSM 本身没有问题，它适合简单流程；但一旦进入真实项目，往往会遇到一个非常典型的场景：

- `idle`、`prepare`、`run`、`finish` 都可能在运行时检测到故障
- 这些状态收到某个错误事件后，都应该跳转到 `fault`
- 如果使用普通 FSM，就只能在每个状态里重复写一遍

也就是说，普通 FSM 很难表达 **这一组状态共享同一种公共规则** 这个语义

### 2.2. HFSM 的层级关系

层级有限状态机(HFSM) 的核心思想是**把共享同一种公共规则的状态组织成一个层级结构**，让公共规则上提到父状态里统一处理；以之前的例子为基础，我们可以把 `idle`、`prepare`、`run`、`finish` 这四个状态组织成一个父状态 `active`，然后在 `active` 里统一处理错误事件：

```text
active
├── idle
├── prepare
├── run
└── finish
```

然后只在 `active` 里处理错误事件，即收到 `ERROR` 类事件时，统一切到 `fault` 状态；这样所有子状态就自动共享了同一套异常处理规则，不再需要逐个复制；因此HFSM 的重点不在于“多套一层树结构”，而在于：

- 可以描述“这一组状态的公共行为”
- 可以把重复逻辑从叶子状态上提到父状态
- 可以让状态图从“平面枚举”变成“带语义的层次结构”

## 3. HFSM 的设计方法

### 3.1. 将状态抽象为类

虽然 C 语言没有类，但我们可以通过结构体和函数指针来模拟 OOP 的状态类；每个状态都可以定义一个结构体，包含 `entry`、`exit`、`handle`、`action` 等成员函数指针；状态之间的层级关系可以通过父子指针来表示；这样就能把状态的行为封装在一个个独立的“类”里，代码结构更清晰，职责更明确；同时也方便后续扩展，例如新增一个状态时，只需要从模拟类中实例化一个新的状态对象，而不需要修改原有的状态逻辑

### 3.2. 事件驱动的状态切换

事件的作用是触发状态变化，而不是代替驱动层命令本身，例如：

- `INIT_COMPLETE`
- `START_CMD`
- `TARGET_READY`
- `MOVE_COMPLETE`
- `TIMEOUT`
- `ERROR_DETECTED`
- `RESET_COMPLETE`

这些事件表达的是：

- 某个阶段满足了推进条件
- 某个异常条件成立
- 某个恢复动作已经完成

它们不负责描述所有实现细节，而只负责把流程从一个阶段推到另一个阶段

### 3.3. 状态的四个核心函数

状态机代码能否长期维护，很大程度上取决于这四类职责是否清楚：

- `entry`：状态切入时的一次性动作，例如初始化变量、发送指令、记录日志等
- `exit`：状态切出时的一次性动作，例如清理资源、停止指令、记录日志等
- `handle`：处理离散事件，例如收到某个命令、检测到某个异常等
- `action`：状态内的周期推进，例如定时检查某个条件、持续监控某个状态等

## 4. HFSM 的设计步骤

### 4.1. 先列举所有阶段

例如一个典型任务流程：

```text
init -> idle -> prepare -> moving -> operating -> finish
                         \-> fault
```

这一步只关注阶段顺序，不关注具体实现细节；同时也不区分一次性动作和周期推进；只要把所有阶段都列出来，形成一个完整的流程图

### 4.2. 找出哪些状态共享公共规则

例如在上面的流程里，`idle`、`prepare`、`moving`、`operating` 这四个状态都可能在运行时检测到故障；因此它们共享同一种公共规则：**如果检测到某个错误事件，就切到 `fault` 状态**；这时就可以把它们组织成一个层级结构，让它们的父状态来统一处理这个公共规则

同时设定一个根阶段 `/`，让所有阶段都从这个根阶段派生出来；这样就形成了一个完整的状态树结构：

```text
/ (root)
├── init
├── active
│   ├── idle
│   ├── prepare
│   ├── moving
│   └── operating
└── fault
```

### 4.3. 定义事件与各阶段处理事件逻辑

例如：

- `INIT_COMPLETE`
- `START_CMD`
- `PREPARE_COMPLETE`
- `MOVE_COMPLETE`
- `OPERATE_COMPLETE`
- `FINISH_COMPLETE`
- `DETECT_ERROR`
- `ERROR_CLEARED`

这样之后，状态机负责“收到事件后如何转移”，而事件何时成立，则交给 `handle` 去判断

### 4.4. 定义 `entry` / `exit` / `action` 逻辑

每个阶段只承担一种职责，entry / exit 负责一次性动作，action 负责周期推进；这样就能把不同职责的代码分开，避免混在一起导致的维护困难

## 5. 使用 X-Macro 来维护状态表

HFSM 的核心是状态表，维护好这张表就能保证整个状态机的正确性；而 X-Macro 则是维护这张表的最佳工具，同时能够解决状态重复性问题；通过 X-Macro 定义状态表，可以把状态的名字、描述、父状态、事件处理等信息集中在一处，避免分散维护带来的同步问题；同时也方便后续扩展，例如新增一个状态时，只需要在 X-Macro 表里添加一行，而不需要修改原有的状态逻辑

例如：

```c
/**
 * @brief 任务状态表
 */
#define STATE_TABLE \
    SX(init, NULL) \
    SX(active, NULL) \
        SX(idle, &state_active) \
        SX(plan_aim, &state_active) \
        SX(moving, &state_active) \
        SX(lasering, &state_active) \
        SX(finish, &state_active) \
    SX(fault, NULL)

/**
 * @brief 任务状态机事件枚举
 */
#define EX(name) MISSION_EVENT_##name,
typedef enum {
    STATE_TABLE
#undef EX

/**
 * @brief 业务状态定义
 */
#define SX(name, parent) \
    static HfsmResult handle_##name(HfsmMachine* m, const HfsmEvent* e); \
    static void entry_##name(HfsmMachine* m); \
    static void exit_##name(HfsmMachine* m); \
    static void action_##name(HfsmMachine* m); \
    static const HfsmState state_##name;
    STATE_TABLE
#undef SX

/**
 * @brief 定义状态结构体
 */
#define SX(n, p) \
    static const HfsmState state_##n = { \
        .name = #n, \
        .parent = p, \
        .handle = handle_##n, \
        .entry = entry_##n, \
        .exit = exit_##n, \
        .action = action_##n \
    };
    STATE_TABLE
#undef SX
```
## 6. 个人编写的 HFSM 库

[点此下载 HFSM 库](/files/hfsm.zip)

目前这个库还在持续迭代中，已经实现的功能有：

- 支持层级状态
- 支持父状态冒泡处理
- 支持 `entry / exit / action / handle`
- 支持事件队列
- 支持最近公共祖先（LCA）转移
- 提供一层额外包装接口，用于约束生命周期

该库分为两层，`hfsm_core` 为核心引擎，`hfsm` 为上层包装接口，简化状态机的构建

其中 `hfsm_config.h` 是用户配置文件，用户可以通过定义宏来调整状态机的行为，例如：

- `HFSM_DEPTH`：最大状态层级深度，默认为 8
- `HFSM_MAX_STATES`：状态数量，默认为 16
- `HFSM_EVENT_DATA_TYPE`：事件数据类型，默认为一个包含指针、整数和浮点数的联合体
- `HFSM_RUN_PARENT_ACTIONS`：是否执行父状态的 `action`，默认为 1（执行）
- `HFSM_ENABLE_ASSERT`：是否启用断言，默认为 1（启用）
- `HFSM_PENDING_QUEUE_MAX`：待处理事件队列的最大长度，默认为 8
- `HFSM_MAX_CHAIN_LENGTH`：事件链条最大长度，必须大于等于 `HFSM_PENDING_QUEUE_MAX`，默认为 `2 * HFSM_PENDING_QUEUE_MAX`

## 6. HFSM 库解析 - `hfsm_core`

### 6.1. 核心对象

从现有实现可以抽象出几个核心对象：

- `HfsmMachine`
- `HfsmState`
- `HfsmEvent`
- `HfsmResult`

它们大致对应下面这些职责

#### 6.2.1. `HfsmMachine`

状态机类，包含：

- 当前状态
- 初始状态
- 事件队列
- 用户上下文指针
- 运行状态标志

#### 6.2.2. `HfsmState`

状态类，包含：

- 状态名
- 父状态指针
- `entry`
- `exit`
- `action`
- `handle`

#### 6.2.3. `HfsmEvent`

状态处理结果，表示：

- `ignore`
- `handled`
- `transition(to_state)`

### 6.3. 事件分发路径

时间分发逻辑表述为：

1. 从当前状态开始调用 `handle()`
2. 如果返回 `HANDLED` 或 `TRANSITION`，停止继续向上
3. 如果返回 `IGNORE`，则沿 `parent` 向上冒泡

即：

- 叶子状态优先处理自身特有事件
- 叶子状态不处理的事件，交给父状态
- 父状态因此可以承接整组子状态的共性规则

### 6.4. 关于 LCA 转移

状态转移并不是简单地把 `current_state` 改成目标状态，而是：

- 从当前状态逐级 `exit`
- 到最近公共祖先停止
- 再向目标状态逐级 `entry`

例如从 `moving` 转到 `lasering`：

- 两者的公共祖先是 `active`
- 因此应退出 `moving`
- 然后进入 `lasering`
- `active` 不应被重复退出与进入

这是 LCA 转移的意义

## 7. HFSM 库解析 - `hfsm`

上层 `hfsm` 主要补充了两类能力：

1. 生命周期约束
2. 状态注册接口

例如：

```c
hfsm.init(&fsm, &ctx);
hfsm.set_initial(&fsm, s_idle);
hfsm.start(&fsm);
```

目的是把一些常见错误拦在更外层：

- 是否已经初始化
- 是否已设置初始状态
- 是否状态数量超限
- 启动之后是否还允许继续修改拓扑

这类约束对于嵌入式工程非常重要，因为业务侧最容易犯的，往往不是“状态机语义写错”，而是“生命周期被破坏”

## 8. 总结

层级有限状态机（HFSM）通过引入层级嵌套与职责分离机制，解决了传统平面 FSM 在复杂嵌入式业务中的逻辑冗余与耦合问题；其核心优势在于利用最近公共祖先（LCA）算法确保了状态切换时动作的精确执行，并借助事件冒泡机制实现了公共业务逻辑的高效收口

在工程实践中，配合 X-Macro 技巧与严密的生命周期管理，HFSM 能够将零散的条件分支转化为高内聚、低耦合的拓扑结构；这不仅提升了系统行为的确定性，也为后续的功能迭代与异常诊断提供了标准化的架构支撑
