## 1. 引言

在使用 C 语言开发控制系统、嵌入式驱动或机器人底层模块时，几乎总会遇到一个问题：

> C 语言没有类、继承、虚函数和访问控制，但工程规模一旦变大，又不可避免地需要“模块化”“解耦”“多实例管理”和“统一入口”

于是，很多项目都会自然地走向“模拟 OOP”这条路，例如：

- 用 `struct + 函数` 模拟对象
- 用函数指针表模拟接口或虚函数
- 用 `void* private` 模拟私有实现
- 用单例模拟全局唯一对象
- 用宏把调用包装成接近成员函数的形式

这些做法本身都不新鲜，问题也不在于“能不能做”，而在于：

- 哪些地方这样做是合理的
- 哪些地方这样做只是让代码看起来像 OOP
- 哪些地方再往前走一步，就会从工程抽象变成维护灾难

真正需要探讨的，不是“C 如何模拟 OOP”，而是：

> **C 语言模拟 OOP 的设计边界应该划在哪里**

---

## 2. C 项目对模拟 OOP 的需求

### 2.1. 项目管理需求

在小工程里，直接写函数通常没有什么问题：

```c
motor_enable(1);
motor_set_speed(1, 10.0f);
```

但当系统逐渐复杂以后，问题开始出现：

-   同一类设备有多个实例
-   真机与仿真需要切换
-   不同厂商硬件需要统一访问入口
-   模块之间不希望直接依赖具体实现
-   初始化顺序和生命周期需要控制
-   全局服务需要唯一入口

这时，单纯的“若干函数”已经不足以表达系统结构，于是就会开始引入：

-   实例对象
-   模块接口
-   单例入口
-   抽象层与实现层分离

也就是说，模拟 OOP 的本质不是为了模仿语言特性，而是为了应对工程复杂度

### 2.2. 模拟 OOP 的收益

如果边界控制得当，C 中适度的 OOP 化通常能带来以下收益：

-   **访问入口更稳定**
-   **上下层依赖更清晰**
-   **多实例管理更自然**
-   **真机 / 仿真 / 不同后端更容易切换**
-   **模块职责更容易划分**

但如果边界控制不好，收益很快就会被副作用吃掉

## 3. 四种对象形态

### 3.1. 入口单例 + 数据单例

入口单例 + 数据单例是最容易理解的一类：

>   系统中确实只存在一个对象，数据和入口都唯一

典型例子：

-   全局日志器
-   全局配置中心
-   系统状态管理器
-   唯一调度器
-   唯一内存池管理器

例如：

```c
extern const struct Logger {
    int level;
    bool(*write)(const char* msg)
} logger_instance;
```

这类设计的前提很明确：

-   语义上只能有一个
-   生命周期覆盖全局
-   不需要多实例扩展
-   上层需要随处访问

#### 数据单例的优点

-   使用简单
-   访问成本低
-   不需要到处传递句柄
-   初始化关系容易集中管理

#### 数据单例的风险

-   容易从“全局服务”退化成“全局变量”
-   任何模块都能依赖它，耦合会扩散
-   测试隔离困难
-   职责很容易越积越多

所以，**数据单例只适合“全局唯一”确实成立的对象**，如果一个对象未来明显可能扩展成多个实例，就不该因为眼下图省事直接做成数据单例

### 3.2. 数据多例

数据多例指的是：

>   系统中存在多个同类对象，每个对象有各自的状态，但访问方式仍然是普通函数 + 显式对象

例如 PID、滤波器、编码器状态、轨迹规划器上下文：

```c
typedef struct {
    float kp;
    float ki;
    float kd;
    float integral;
    float prev_err;
} Pid;

void pid_init(Pid* self, float kp, float ki, float kd);
float pid_calculate(Pid* self, float target, float actual, float dt);
```

调用时：

```c
Pid joint1_pid;
Pid joint2_pid;

pid_init(&joint1_pid, 1.0f, 0.1f, 0.01f);
pid_init(&joint2_pid, 1.2f, 0.1f, 0.02f);
```

这类模式最朴素，但往往也是最健康的

#### 数据多例的适用场景

-   算法模块
-   状态对象
-   小型控制器
-   不需要运行时切换后端
-   不需要统一全局入口

#### 数据多例的优点

-   结构简单
-   IDE 友好
-   调试路径直接
-   不需要动态分发
-   没有额外的全局状态

#### 数据多例的边界

对于这类模块，通常**没有必要强行往 OOP 方向继续推进**，例如 PID、低通滤波器、状态估计器，本质就是“参数 + 状态 + 计算函数”，继续加接口单例、方法表、`void* private`，收益通常很低

### 3.3. 入口单例 + 数据多例

在数据多例的基础上统一入口，由入口单例统一管理，其核心不是“对象唯一”，而是：

>   **访问入口唯一，但底层数据是多实例**

也就是：

-   上层只认一个统一入口
-   入口背后持有一套抽象接口
-   每个设备实例仍然有各自的数据和状态
-   真机 / 仿真 / 厂商差异被收进统一入口之后

典型例子是电机、总线、执行器、传感器管理层

例如：

```c
#define motor motor_ops_instance

typedef struct {
    uint16_t id;
    float pos;
    float vel;
    float torque;
} Motor;

typedef struct {
    int(*enable)(Motor* self);
    int(*disable)(Motor* self);
    int(*set_torque)(Motor* self, float tau);
    int(*update)(Motor* self);
} MotorOps;

extern const MotorOps* motor_ops_instance;
```

调用时：

```c
extern Motor motor1;
extern Motor motor2;

motor.enable(&motor1);
motor.set_torque(&motor2, 0.5f);
```

#### 这种模式的核心价值

它解决的是“**实现层可替换，但实例状态不该被做成全局唯一**”的问题

例如：

-   同一套业务逻辑，需要在真机和仿真之间切换
-   同一套上层控制代码，需要兼容不同厂商电机
-   不同实例共享同一套驱动实现，但各自状态独立
-   入口需要稳定，但对象数量显然不止一个

换句话说，它不是在模拟“类”，而是在解决：

-   **统一抽象入口**
-   **后端实现替换**
-   **实例数据独立管理**
-   **减少上层分支判断**

### 3.4. 入口多例 + 数据多例

入口多例 + 数据多例指的是：

>   系统中不仅底层数据是多实例，连访问入口本身也是多实例

也就是每个对象不仅有自己的状态，还持有自己的一套操作入口，典型形式通常是：

-   每个实例内部保存一组函数指针
-   不同实例可以绑定不同实现
-   调用时通过对象自身入口完成分发
-   数据与操作入口一起构成完整对象

例如：

```c
typedef struct {
    int(*enable)(Motor* self);
    int(*disable)(Motor* self);
    int(*set_torque)(Motor* self, float tau);
    int(*update)(Motor* self);
    uint16_t id;
    float pos;
    float vel;
    float torque;
} Motor;
```

调用时：

```c
extern Motor motor1;
extern Motor motor2;

motor1.enable(&motor1);
motor2.set_torque(&motor2, 0.5f);
```

这类模式看起来最接近传统 OOP，因为每个对象都同时包含：

-   自己的数据
-   自己的操作入口
-   自己绑定的具体实现

从抽象表达上看，它确实最完整，但在 C 语言中通常也是最容易越界的形态

#### 入口多例 + 数据多例的适用场景

-   确实存在对象级别的动态多态需求
-   不同实例需要在同一时刻绑定不同后端实现
-   统一入口单例已经无法表达对象间实现差异
-   上层确实需要按对象维度切换行为

#### 入口多例 + 数据多例的优点

-   抽象表达最完整
-   不同实例可以独立绑定不同实现
-   单个对象的行为与状态绑定更直接
-   某些插件式或驱动适配场景下具有灵活性

#### 入口多例 + 数据多例的风险

-   每个实例都携带一份入口信息，对象布局更重
-   调用路径变长，IDE 跳转和阅读体验明显变差
-   很容易引入 `void* private`、强转链和生命周期混乱
-   一旦没有真正的对象级多态需求，这种设计就会变成过度抽象

因此，这种模式虽然最像 OOP，但在 C 语言中通常并不推荐作为默认方案。只有当“**不同实例必须在同一时期绑定不同实现**”这一点真实成立时，才值得引入；否则，大多数场景使用“入口单例 + 数据多例”已经足够，再继续推进往往只会增加复杂度而不会带来成比例的收益

## 4. “入口单例 + 数据多例”的优点

### 4.1. 比“入口单例 + 数据单例”更不容易锁死扩展性

如果把所有电机都做成一个全局单例：

```c
extern Motor g_motor;
```

那么一开始也许很方便，但很快会遇到问题：

-   多电机系统无法自然扩展
-   同类资源无法复用
-   实例状态混在一起
-   接口与对象语义不一致

这说明：

>   **设备类对象往往不该做成入口单例 + 数据单例**

### 4.2. 比“入口多例 + 数据多例”更克制

入口多例 + 数据多例的问题在于：

-   代码阅读要先看结构体，再全局搜索函数指针指向
-   IDE 跳转不友好
-   `private` 强转链很痛苦
-   每个实例都携带方法表，布局更重
-   生命周期与所有权更容易混乱

如果底层实现对同类对象其实是统一的，那这套 per-instance 方法表通常只是多余复杂度

而“入口单例 + 数据多例”的做法把方法表从每个对象身上剥离掉，只保留一份：

-   **接口唯一**
-   **实例轻量**
-   **切换后端集中**
-   **数据仍然多例**

这就是它最关键的工程价值

### 4.3. 性能优化与开销控制

这里的“性能提升”并非单纯指代控制算法执行速度的上限，而侧重于通过**降低抽象开销**优化工程质量：

-   **减少每个实例化对象的函数指针表开销，降低对象布局的复杂度**
-   **减少运行时的动态分发层数，尽可能使编译器在编译期完成类型确定，从而利用内联优化**
-   **确保同类对象共享统一的逻辑入口，提升指令缓存的命中率**
-   **在缩减二进制代码尺寸的同时，通过结构解耦降低长期维护的认知负担**

总之，该设计优先提升的是**工程维度的健壮性与可扩展性**，其次才是由于减少了间接寻址所带来的局部运行效率收益

------

## 5. 四种模式的边界

这四种最容易混淆，所以需要有一套清晰判断方法

### 5.1. 什么时候用入口单例 + 数据单例

满足以下条件时，可以考虑数据单例：

-   系统中语义上只能有一个
-   生命周期天然覆盖全局
-   不会扩展为多个同类实例
-   被多个模块共享访问是合理的
-   作为服务中心比作为业务对象更合适

例如：

-   logger
-   config
-   global scheduler
-   diagnostic center

#### 不适合数据单例的信号

-   已经出现多个设备 ID
-   未来很可能支持多通道 / 多节点
-   当前只是图方便，不是真正唯一
-   把“唯一入口”和“唯一对象”混为一谈

如果对象本来应当多例，只是上层想偷懒不想传句柄，这不是做单例的理由

### 5.2. 什么时候用数据多例

满足以下条件时，优先考虑数据多例：

-   每个实例有独立状态
-   模块主要是算法或状态容器
-   不需要统一切换后端实现
-   上层显式传递对象是可接受的
-   实例之间没有必要共享全局入口

例如：

-   PID
-   Kalman
-   LPF
-   状态机上下文
-   插值器
-   运动学求解上下文

#### 数据多例的原则

这类模块要尽量克制，通常做到：

-   `struct` 表示对象
-   普通函数表示操作
-   最多用少量宏做语法糖
-   不引入接口单例
-   不引入 `void* private`
-   不引入虚函数式分发

这往往就是最好的边界

### 5.3. 什么时候用入口单例 + 数据多例

当一个模块同时满足以下特点时，这种模式最合适：

-   上层希望统一访问入口
-   同类对象存在多个实例
-   后端实现存在替换需求
-   真机 / 仿真 / 平台差异需要隔离
-   每个对象的状态仍然必须独立

例如：

-   motor
-   can device
-   sensor backend
-   actuator driver
-   storage backend
-   communication adapter

#### 这种模式的判断关键

不是看它“像不像类”，而是看它是否同时存在：

1.  **多实例状态**
2.  **统一抽象入口**
3.  **整批实现替换需求**

三者缺一，通常都没必要上这一层

------

## 6. 容易越界的几个地方

边界一旦没收住，代码就很容易开始恶化

### 6.1. 为了像成员函数，强行给每个对象挂方法表

这类写法的最大问题不是“不能用”，而是**收益和代价不成比例**

如果一个模块根本不需要对象级动态多态，那么：

-   方法表会增加阅读成本
-   跳转定义变得困难
-   AI 辅助也容易找错实际实现
-   每个实例都平白背上一层间接调用

尤其在嵌入式场景里，这种成本往往并不值得

### 6.2. 为了模拟 private，滥用 `void* private`

`void* private` 最大的问题不是语法，而是它会模糊边界：

-   所有权不清晰
-   真实类型不清晰
-   强转链很痛苦
-   一旦嵌套多层，理解成本暴涨
-   调试器看到的只是裸地址

如果模块只是想隐藏少量内部字段，很多时候更好的做法其实是：

-   直接在 `.c` 中隐藏实现细节
-   或者把结构体分成公开配置与内部状态
-   或者使用句柄，但不要层层嵌套继承幻觉

#### 一个很重要的判断

>   如果 `private` 的存在只是为了“看起来像类”，那通常已经越界了

### 6.3. 把所有模块都抽成统一接口风格

接口统一并不等于设计统一

例如：

-   PID 是算法状态模块
-   Motor 是设备抽象模块
-   Logger 是全局服务模块

这三者虽然都能硬套成“对象 + 方法表”，但它们本质上不是同一种问题

真正合理的设计应该是：

-   算法模块，用数据多例
-   全局服务，用数据单例
-   可替换后端的设备类模块，用入口单例 + 数据多例

如果为了风格一致，把所有模块都强行抽成同一套模拟 OOP 形式，最后通常会变成形式统一、复杂度统一、痛苦也统一

### 6.4. 让抽象层只增加写法，不减少复杂度

这是最关键的一条

任何抽象都应该回答一个问题：

>   **它有没有减少系统的真实复杂度？**

如果一个抽象只带来了这些效果：

-   代码看起来更像 C++
-   调用语法更像成员函数
-   命名更“高级”
-   结构图更漂亮

但没有真正减少：

-   分支扩散
-   后端耦合
-   重复代码
-   生命周期混乱
-   模块边界不清

那么它就不是工程抽象，而只是语法装饰

## 7. 划界方法

比起讨论“是否 OOP”，更好的办法是问下面几个问题：

-   **对象是否天然唯一：** 如果答案是“是”，优先考虑数据单例
-   **对象是否天然多例：** 如果答案是“是”，不要为了省传参就把它做成单例
-   **是否存在整批实现替换需求：** 如果答案是“是”，考虑入口单例
-   **实例状态是否必须独立：** 如果答案是“是”，数据必须多例
-   **模块是否只是算法状态容器：** 如果答案是“是”，不要引入运行时方法表
-   **抽象是否改善了阅读与维护：** 如果答案是“否”，说明已经越界

------

## 8. 总结

C 语言当然可以模拟 OOP，但真正值得追求的，从来不是表面语法，而是工程边界

如果边界划得合理：

-   单例可以让全局服务更清晰
-   多例可以让状态管理更自然
-   入口单例 + 数据多例可以在统一抽象入口的同时，保留实例独立性和实现可替换性

如果边界划得不好：

-   单例会变成全局依赖扩散器
-   多态会变成函数指针迷宫
-   private 会变成类型不透明的强转地狱
-   模拟 OOP 会变成“看起来高级，维护起来痛苦”的屎山

所以，关于 C 语言模拟 OOP 的设计边界，最重要的结论其实很简单：

>   **不要为了模仿类而抽象，而要为了控制复杂度而抽象**

当某种写法真的让模块职责更清楚、依赖更稳定、后端更容易切换时，它就是合理的；当某种写法只是让代码更像 OOP，却没有减少任何真实复杂度时，它就已经越界了
